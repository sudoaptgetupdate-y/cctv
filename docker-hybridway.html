1. การเปรียบเทียบสภาพแวดล้อม

เหตุผลที่เราใช้ Docker เฉพาะบน Production

💻

Local PC (Development)

  • ✅ เน้นความเร็วในการแก้โค้ด (Hot Reload)
  • ✅ รัน `npm run dev` เพื่อเห็นผลทันที
  • ✅ ใช้เครื่องมือ Debugging ของ OS ได้เต็มที่
🐳

Production (192.168.123.123)

  • ✅ สภาพแวดล้อมคงที่ (Immutable)
  • ✅ แยกบริการ (Isolation) ไม่ให้ตีกัน
  • ✅ Auto-restart หากระบบล่ม (Self-healing)

2. ขั้นตอนการ Deploy (Hybrid Workflow)

🛠️

Step 1: Local Dev

พัฒนาบน Local เมื่อพอใจแล้วสั่ง Push โค้ดขึ้น Git

git add .
git commit -m "update cctv"
git push origin main
☁️

Step 2: Server Pull

SSH เข้าเซิร์ฟเวอร์ และดึงโค้ดล่าสุดลงมา

ssh user@192.168.123.123
cd /cctv-project
git pull origin main
⚡

Step 3: Hybrid Build

ใช้ Docker Compose สั่ง Build และรันใหม่ทันที

docker-compose up -d --build

Docker จะใช้ Cache ส่วนที่ไม่ได้เปลี่ยน ทำให้รันเร็วมาก!

💡 ทำไมต้องใช้ `--build` ทุกครั้ง?

เมื่อเราเลือกใช้ "Hybrid Way" (Git Pull -> Build on Server) การใช้ Flag --build จะบอกให้ Docker ตรวจสอบว่ามีไฟล์ไหนในโฟลเดอร์เปลี่ยนไปบ้าง

ข้อดี (Pro)
  • • ไม่ต้องตั้ง Registry กลาง
  • • แก้ไข Bug หน้างานได้เร็ว
  • • ประหยัด Bandwidth ขาออก
ข้อควรระวัง (Caution)
  • • Server ต้องมี CPU/RAM พอสำหรับ Build
  • • ครั้งแรกอาจจะนานเพราะต้องโหลด Base Image
*เหมาะมากสำหรับการพัฒนาระบบที่มีการอัปเดตบ่อย (Agile) และรันบนระบบ Network ภายในตำบล
