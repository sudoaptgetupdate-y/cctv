# 🎥 CCTV Streaming Technical Knowledge Base

เอกสารฉบับนี้รวบรวมเกร็ดความรู้และคำอธิบายทางเทคนิคเกี่ยวกับระบบ Video Streaming ที่ใช้งานในโปรเจคนี้ เพื่อความเข้าใจในการพัฒนาและปรับจูนประสิทธิภาพ

## 1. Video Codecs (มาตรฐานการบีบอัดวิดีโอ)

### **H.264 (AVC)**
*   **คืออะไร:** มาตรฐานการบีบอัดวิดีโอที่ได้รับความนิยมสูงสุด
*   **ทำไมถึงใช้:** 
    *   Browser ทุกตัวรองรับ (Native Support)
    *   รองรับ Hardware Acceleration (ถอดรหัสผ่านการ์ดจอได้โดยตรง ไม่กิน CPU)
    *   ความหน่วงต่ำ (Latency ต่ำ)
*   **สถานะในระบบ:** เราใช้เป็นมาตรฐานหลัก (Stable Feed)

### **H.265 (HEVC)**
*   **คืออะไร:** มาตรฐานใหม่ที่บีบอัดได้ดีกว่า H.264 เท่าตัว (ประหยัดแบนด์วิดท์มากขึ้น)
*   **ข้อจำกัด:** Browser หลายตัวยังไม่รองรับการเล่นสดผ่านเว็บโดยตรง หรือต้องการเครื่องที่มีสเปคสูง
*   **วิธีแก้ในระบบ:** หากกล้องเป็น H.265 เราจะใช้ระบบ **Transcoding (FFmpeg)** เพื่อแปลงเป็น H.264 ก่อนส่งให้ผู้ใช้งาน

## 2. Streaming Modes & Protocols

ในโปรเจคนี้เราใช้เทคโนโลยีจาก **go2rtc** ซึ่งรองรับการสลับโหมดอัตโนมัติเพื่อให้ดูได้ในทุกสภาพเครือข่าย:

### **WebRTC (Real-Time Communication)**
*   **จุดเด่น:** ความหน่วงต่ำที่สุด (Ultra-Low Latency < 0.5 วินาที)
*   **การทำงาน:** ส่งข้อมูลแบบ UDP (Peer-to-Peer) เหมาะสำหรับการควบคุมกล้อง (PTZ) หรือการดูเหตุการณ์สดที่ต้องการความเร็วสูง
*   **ข้อจำกัด:** อาจถูกบล็อกโดย Firewall หรือ VPN บางประเภท

### **MSE (Media Source Extensions)**
*   **จุดเด่น:** ความเสถียรสูงสุด (Stable) และความหน่วงต่ำ (1-2 วินาที)
*   **การทำงาน:** ใช้ JavaScript ช่วยจัดการชิ้นส่วนวิดีโอและส่งให้ HTML5 Video Player
*   **ข้อจำกัด:** ไม่รองรับใน iPhone/iPad (iOS) เวอร์ชันเก่า (ต้องใช้ HLS แทน)

### **HLS (HTTP Live Streaming)**
*   **จุดเด่น:** ดูได้ทุกที่ ทุกอุปกรณ์ แม้อินเทอร์เน็ตจะช้าหรือแกว่ง
*   **การทำงาน:** แบ่งวิดีโอเป็นไฟล์เล็กๆ (.ts) แล้วส่งผ่าน HTTP ธรรมดา
*   **ข้อจำกัด:** ความหน่วงสูง (5-10 วินาที)

## 3. ระบบ Hybrid Fallback Strategy (ในโปรเจคนี้)

เราตั้งค่าใน `WebRTCPlayer.jsx` โดยใช้ Parameter `mode=webrtc,mse` ซึ่งหมายถึง:

1.  **Priority 1 (WebRTC):** ระบบจะพยายามเชื่อมต่อด้วย WebRTC ก่อนเพื่อให้ได้ภาพที่เร็วที่สุด
2.  **Priority 2 (MSE):** หากภายใน 1-2 วินาที WebRTC เชื่อมต่อไม่สำเร็จ (เช่น ติด Firewall) ระบบจะสลับไปใช้ MSE อัตโนมัติเพื่อให้ภาพติดชัวร์
3.  **Priority 3 (HLS/MP4):** หาก MSE ไม่รองรับ (เช่น บน iOS) ระบบจะถอยไปใช้โหมดที่เข้ากันได้กับอุปกรณ์นั้นๆ

## 4. ตารางเปรียบเทียบประสิทธิภาพ

| โหมดสตรีมมิ่ง | ความหน่วง (Latency) | ความเสถียร | การรองรับอุปกรณ์ |
| :--- | :--- | :--- | :--- |
| **WebRTC** | ⚡ < 0.5 วินาที | ปานกลาง | PC/Android |
| **MSE** | 🔵 1-2 วินาที | ✅ สูงมาก | PC/Android |
| **HLS** | 🐢 5-10 วินาที | ✅ สูงมาก | ทุกอุปกรณ์ (รวม iOS) |

## 5. คำศัพท์ที่พบบ่อย (Glossary)

*   **Main Stream (HD):** สตรีมหลักจากกล้อง ความละเอียดสูง (1080p/4K) เหมาะสำหรับการขยายดูรายละเอียด
*   **Sub Stream (SD):** สตรีมรอง ความละเอียดต่ำ (360p/480p) เหมาะสำหรับการดูหลายกล้องพร้อมกันเพื่อประหยัด CPU และเน็ต
*   **Transcoding:** การนำวิดีโอมาเข้ารหัสใหม่ในจังหวะที่ส่ง (เช่น แปลง H.265 เป็น H.264) ใช้ทรัพยากร Server สูงแต่ช่วยให้เปิดดูได้ทุกที่
*   **Pass-through:** การส่งข้อมูลวิดีโอจากกล้องไปหาผู้ใช้ตรงๆ โดยไม่แก้ไข ประหยัดพลังงาน Server ที่สุด
